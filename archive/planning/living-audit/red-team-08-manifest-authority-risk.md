## Planning slice challenged

Adversarial review of `existing-runtime-adoption-policy.md`: whether the proposed inert-adoption model is safe enough without a stronger `workframe-manifest.json` authority and provenance contract.

## Current-state facts

- Public install docs list `workframe-manifest.json` as part of the generated project layout and tell users to run lifecycle CLI commands from the project root where the manifest lives.
- The installer currently writes `workframe-manifest.json` near the end of scaffold generation.
- `parseArgs()` turns a positional project name into non-interactive defaults, sets `force = true`, allows install actions, and pulls Docker unless `--ci` is set.
- `prepareTarget()` removes an existing target directory when `force` is true.
- The generated manifest records scripts, native agent identity, Docker names, ports, layout, integrations, and a small security section.
- The manifest does not yet appear to encode enough provenance to authorize update/open/connect decisions: source commit, package tarball identity, generated-file digests, data path classification, backup references, or last package-truth evidence.
- Current docs and planning correctly say host Hermes is off limits, only managed Docker Hermes is executable in v0.1.x, and non-Hermes host runtimes are inert candidates.

## Red-team contradictions / weak assumptions

### 1. The manifest exists, but is not yet an authority

The previous red-team pass treated manifest authority as under-specified. The installer already writes a manifest, but that does not solve the authority problem. A file named `workframe-manifest.json` can be stale, hand-edited, copied between folders, or generated by an older package.

Safer distinction:

```text
manifest_present != cell_authority
cell_authority = manifest + provenance + ownership + health + negative-write guarantees
```

### 2. Default positional install currently behaves like destructive create

The command the product wants users to run is `npx create-workframe MyBusiness`. In current installer parsing, a positional name implies `--yes`, `--force`, `--allow-install-actions`, and Docker pull behavior. Combined with `prepareTarget()`, the default happy-path command can remove an existing same-named directory.

This is not merely an implementation issue; it contradicts the planning rail's non-destructive adoption posture. The first-principles rule should be: the most memorable command must be the safest command.

### 3. Adoption policy assumes the cell boundary is already trustworthy

The adoption policy correctly separates `host_tool_marker`, `managed_runtime`, and `adapter_binding`. But it assumes Workframe can first identify a managed cell safely. If manifest authority is weak, runtime-adoption safety is built on sand.

Bad shortcut to forbid:

```text
folder_has_manifest -> safe_to_open_or_adopt
folder_has_Agents -> Workframe-owned runtime
container_names_match -> safe managed runtime
```

### 4. The manifest can become a secret or privacy leak

A manifest should identify cell structure without exposing users. Current and planned fields can leak project names, profile slugs, install paths, Docker stack names, ports, remote URLs, and deployment posture. Runtime detection planning already recognizes privacy risk; manifest design needs the same standard.

Redaction rule: manifest is local authority, not shareable support bundle. Any export should use a separate redacted diagnostic report.

### 5. Generated lifecycle scripts can outgrow manifest truth

Generated scripts include setup, start, UI open, update Hermes, profile bootstrap, chat, and doctor paths. If future updates edit scripts without recording provenance and expected digests, `open` and `doctor` may report confidence on a folder whose scripts no longer match the package that created it.

Minimal gate:

```text
script exists -> not enough
script digest matches manifest or package evidence -> eligible for trusted lifecycle operation
```

### 6. Native deploy vocabulary conflicts with the inert-host-runtime policy

The installer has a `--deploy native` path that detects host Hermes and writes a native-Hermes note while still using isolated `./Agents` for Docker. Planning now says host Hermes is marker-only and non-executable for v0.1.x. The language around `native` risks being interpreted as host-runtime adoption.

Simpler rule: remove `native` from release claims until it is renamed or scoped as `detect_host_hermes_marker_only`.

## Simpler first-principles alternative

Before adapter adoption, define a minimal `CellAuthorityGate`:

```text
CellAuthorityGate
  manifest_schema_version
  cell_id
  created_by_package_name
  created_by_package_version
  created_by_installer_blob_or_commit
  generated_at
  install_mode
  declared_data_paths
  declared_runtime_paths
  expected_generated_file_digests
  current_health_snapshot
  last_package_truth_gate_reference
  secrets_absent_from_manifest
```

Then require:

```text
create: target must not exist, unless an explicitly destructive verb is added later
open: CellAuthorityGate passes, then read-only health/status
update: CellAuthorityGate + backup + dry-run migration + rollback evidence
connect: remote CellAuthorityGate plus displayed execution/credential boundary
adopt runtime: CellAuthorityGate + redacted candidate + explicit consent + adapter harness
```

## Security issues to pin before implementation

1. The default `npx create-workframe MyBusiness` command must not delete an existing same-named folder.
2. Manifest fields must never include raw secrets, token paths, auth cache paths, or provider account identifiers.
3. Raw manifest should not be the support artifact; produce a redacted diagnostic report instead.
4. Host runtime detection must not write to the manifest until the user explicitly binds a validated adapter.
5. `native` deploy language must not imply host Hermes is supported as a Workframe runtime.
6. Public/VPS mode must not trust a local desktop manifest without server-side authority checks.
7. Update and repair flows must require a backup reference before touching `Agents/`, `Files/`, vault data, compose files, or generated scripts.

## Proposed migration / refactor steps

1. Add a planning-only `CellAuthorityGate` spec before any implementation of open/update/connect/adoption.
2. Change release-readiness criteria to require existing-target refusal for the default positional install command.
3. Split raw `workframe-manifest.json` from `workframe-diagnostic-redacted.json` in planning.
4. Add manifest provenance fields to the next planning pass: package identity, source commit/blob, generated file digest map, and last evidence reference.
5. Reclassify `--deploy native` as unsupported/deferred in v0.1.x release language unless direct code review proves it is harmless and non-adopting.
6. Require doctor/open tests to prove no writes occur to `.env`, compose files, `Agents/`, `Files/`, vault, routes, generated scripts, or manifest.
7. Require all runtime-adoption plans to start with `CellAuthorityGate` before `RuntimeInventoryReport`.

## Diagnostic conclusion

The strongest current risk is not the adapter matrix itself. It is that Workframe may treat generated folder shape or a weak manifest as sufficient proof of ownership, then build open/update/connect/adoption behavior on top of that weak boundary. The next convergence pass should define manifest authority and package provenance before expanding surfaces or runtime detection.

## Next best planning target

Cell manifest and provenance authority: specify `workframe-manifest.json`, redacted diagnostics, existing-target refusal, generated-file digests, and read-only `doctor/open` guarantees.
